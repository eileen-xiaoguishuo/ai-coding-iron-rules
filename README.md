# AI 编程铁律

> 我是 **Eileen**。20 多年的产品老兵，一家体育赛事系统公司的老板，不是科班程序员。
> 用 AI（Claude Code）多窗口并行，和它共创出真实在跑的产品。
>
> 这里是我「AI 共创日记」里沉淀下来的**铁律**——不是教程，是战场。

每一条都来自一次真实翻车：我先信了 AI 的"看起来对"，结果绕了好几圈，
最后把教训钉成一条规矩，下次不再犯。如果你也在用 AI 写代码，
希望这些能帮你少走我走过的弯路。

## 一、心法：怎么"信"AI

### 1. 用证据说话，不臆测
**坑**：AI 说"应该修好了""可能是缓存问题"，我信了，结果一跑全错，来回绕好几轮。
**律**：任何判断先拿证据——跑命令、看日志、读文件、实际复现。说"完成 / OK"之前**必须**验证过。禁止"我觉得 / 大概 / 应该 / 看起来"这种没有证据的裸主张。
**为什么**：AI 最擅长给"听起来对"的答案。它的自信不等于事实。这是其他所有铁律的根。

### 2. 结果对 ≠ 过程对：让它交代"怎么做的"
**坑**：AI 给了个看着没问题的结果，我验了一处是对的就放心了，后面一连串都是错的，靠的是巧合蒙对的。
**律**：让 AI 在给结果时主动说清三件事——**用了什么 / 依据什么 / 怎么验证的**。尤其涉及"有多个选项"（用哪个字段、哪个文件、哪个接口）时。
**为什么**："结果碰巧对"会把错误的实现方式藏起来，等你发现时已经错了一片。过程透明才查得动。

### 3. 你的业务直觉 > AI 的"看起来合理"
**坑**：AI 顺着数据推出一个结论，我第一反应是"这不可能"，但被它一堆"也许、可能"说服了，最后证明我的直觉是对的。
**律**：当你凭对业务的了解觉得"不对劲"时，让 AI **立刻停止揣测**，别再帮它找"也许成立"的理由。先回到事实。
**为什么**：AI 看的是代码和数据模式，你看的是真实世界。涉及业务事实，你的记忆 > 它的"合理推测"。

## 二、怎么给 AI 派活

### 4. 指令一模糊，先给选项，别让它猜
**坑**：我说"优化一下""简化下""把那个删掉"，AI 自作主张选了个方向，干完发现不是我要的，白忙。
**律**：边界模糊的指令（先下线 / 改一下 / 删掉 / 对齐 / 整理…），让 AI **先停下来列 2–4 个具体方案**给你选，再动手。
**为什么**：模糊指令下 AI 会挑一个它觉得最可能的去做，猜错的成本由你买单。先对齐再开工最省时间。

### 5. 警惕 AI 的"自动化偏好"：先看真实频率
**坑**：一个一周才用几次的小事，AI 兴冲冲给我上了个每小时跑的定时任务，纯属浪费。
**律**：它默认推荐"自动化 / 加缓存 / 加抽象"时，先问一句**真实使用频率/规模有多大**。低频的事，手动一个按钮就够。
**为什么**：AI 被训练得偏爱"看起来完善"的方案，容易过度工程。好取舍要看实际需求，不是看哪个"更全"。

### 6. 想听真话，就让它当"红队"
**坑**：我问 AI"这个方案行不行"，它几乎总说"很好！"，附和完我才发现方案有大窟窿。
**律**：要评审时明确要求——"挑刺、不要附和，给我至少 3 个具体质疑，指出最弱的一环"。讨论阶段明确说"先别动手"。
**为什么**：AI 默认倾向讨好你、推进任务。你不主动要求对抗，就听不到真问题。

---

### 7. 并行协作：开工前先确认没人在做同一件事

**坑**：我同时开着好几个工作流（可以理解成几个人、或几个 AI agent 并行）。两个流各自把同一个功能做了一遍——重复劳动还算小事；更糟的是其中一个已经做完、上线了，另一个不知情，还基于"线上没有这个功能"的判断，差点用自己的旧版本覆盖掉已上线的新版本（直接退化）。

**律**：并行协作时，开工做一个"可能和别人撞车"的功能前，先花一分钟确认没人正在做、或已经做完——翻共享的进度记录、看提交历史、查线上的真实现状。尤其判断"线上到底有没有"，要去查线上真实状态，不能因为"我本地没有"就断定"线上没有"。

**为什么**：并行能提效，代价是信息不同步。重复劳动只是浪费时间；拿旧版覆盖别人已上线的新版，是实质性的退化和回归。"我这儿没有"从来不等于"哪儿都没有"——本地的缺失不代表全局的缺失。

## 三、调试：出问题先查哪

### 8. "改了没生效"第一步：确认你测的是不是新版本
**坑**：我明明改了代码，却没生效，怀疑了半天代码——结果是我测的还是旧的编译产物 / 旧进程 / 连错了环境。
**律**：遇到"改了表现没变"，**第一步不是看代码，是确认你正在测的就是新版本**：编译产物的时间戳新不新？跑的是不是新进程？连的是哪个环境？
**为什么**：一大半"代码没改对"的错觉，其实是你在对着旧东西验证。先排除这个，能省掉无数无效 debug。

### 9. 别闷头看代码，先看运行时
**坑**：界面没渲染 / 数据不对 / 接口报错，我一头扎进代码逻辑里硬读，读半天。
**律**：这几类现象，**第一步先看运行时状态**——日志（重载时间戳）、真实数据、浏览器 Network 面板、直接 curl 接口。
**为什么**：现象往往不在你以为的那段逻辑里。运行时的真相比静读代码快十倍。

### 10. "解析出 0 条"先看原始数据长啥样
**坑**：导入 / 解析报"0 条、没解析到"，我先怀疑业务逻辑，其实是原始数据的形态跟我以为的不一样（被编码过、表头多了一行、列名换了说法）。
**律**：解析报空，**第一步 dump 一条原始数据出来看**：是不是被编码了？真正的表头在第几行？字段名是不是变体？严禁 `except: continue` 把解析错误悄悄吞掉。
**为什么**：脏在源头时，怀疑逻辑就是南辕北辙。先看清输入，再谈处理。

### 11. 别只信表面信号（状态码 / 镜像 / 缓存）
**坑**：我用状态码判断"接口存在不存在"，结果鉴权层在路由匹配之前就拦截返回了，存在与否都是同一个码，骗了我。
**律**：判断时去查**真正生效的权威源**，别信那些会过期、会被中间层改写的旁路信号（状态码、本地镜像、缓存、界面显示）。
**为什么**：表面信号经常和底层真相不一致。卡住时，回到权威数据。

## 四、交付：说"做好了"之前

### 12. 说"做好了"之前，自己先验证一遍
**坑**：AI（和我）说"可以测了"，结果用户一点就白屏 / 报错，把对方当成了测试机。
**律**：交付前**自己先走一遍真实流程**。前端尤其要确认：真打开过、没白屏、关键路径点得通。API 层的"测试通过"不等于用户视角通过。
**为什么**：没自测的"完成"是在拿别人的时间赌。验证一次的成本，远小于让对方踩坑再回来。

### 13. "我这儿是好的"≠"用户那儿是好的"
**坑**：用户截图说有问题（有条缝 / 错位 / 显示不对），我本地怎么都复现不出来，俩人各说各话。
**律**：复现不出时，**按用户的真实环境来**——他的屏幕缩放比例、设备、浏览器。视觉问题用工具放大像素确认，别靠"我看着没问题"。
**为什么**：缩放、DPI、设备差异会让同一份代码在两边表现不同。以用户看到的为准。

### 14. 危险操作的默认值必须是"什么都不做"
**坑**：合并 / 删除 / 批量修改类的工具，默认值设成了"执行"，手一滑就误操作。
**律**：删除、合并、去重、批量改这类危险动作，**默认值一律=不做事**，让人每次主动勾选才执行。
**为什么**："手滑误删"必须要费点劲才会发生。把破坏性操作设成需要主动确认，是给未来的自己买保险。

## 五、数据与安全

### 15. 外部数据入库前先洗干净：源头脏，下游全脏
**坑**：拿一份外部表格（Excel / 抓来的数据）直接入库，结果脏数据污染了一整片，进了库再查就是拆东墙补西墙。
**律**：把外部数据当权威源入库**之前**，先出一份"体检报告"修干净（合并单元格、格式错乱、错别字、离谱值）。入库做成幂等的，重跑只纠正不重复。
**为什么**：源头一脏，下游全脏，而且回溯不到是哪一格出的问题。在门口拦住，比进屋后打扫便宜得多。

### 16. 跨来源比字符串，先归一化（空格 / 大小写）
**坑**：两份数据按名字对应，怎么都对不上——其实只差一个空格 / 大小写，被严格相等判成了"不匹配"，整批漏掉。
**律**：跨系统比字符串前**必先归一化**：去空格、统一大小写，再比。别裸用 `==`。
**为什么**：外部数据里的脏空格、大小写差异极常见。比对卡死时，十有八九是没归一化。

### 17. 凭证密钥：绝不写默认值，绝不进仓库
**坑**：读密钥写成"取不到就用一个默认值"，那个默认 secret 跟着代码被推上了仓库，躺了一个多月没人发现。
**律**：密钥 / 密码 / token 必须**没配就报错**（fail loud），绝不给默认值兜底。任何要公开的代码，先扫一遍有没有夹带凭证。
**为什么**：默认值会悄悄变成线上配置；密钥进了公开仓库，删了也可能已被爬走。**正打算把代码分享到 GitHub 的你，这条尤其要记牢。**
